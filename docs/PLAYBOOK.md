# 工程规范 Playbook（REK 内核）

## 0. 目标
两人团队默认目标：**极快交付 + 安全稳定 + 易维护**  
优先级：减少返工 > 并行不卡 > 可定位 > 性能优化（除非明确性能目标）。

---

## 1. 产物与流程（工具无关）
默认工作流（最小闭环）：
1) RFC（需求：锁边界/口径/验收）
2) ADR（关键决策：超短记录；无决策则跳过）
3) INS（执行：范围+计划+验证）
4) 实现 + 测试
5) PR 合并（保持主干可发布）

说明：
- **INS 不重复全局红线**，只写“本任务范围/计划/验证”，其余引用本 Playbook。

---

## 2. 并行开发规则（避免互相卡）
- 契约先行：接口/数据结构/错误语义先确定，再并行实现
- 小批量合并：尽量当天可合并；避免长分支
- 核心模块改动需另一人 Review（核心模块由项目自行定义并维护列表）

---

## 3. 风险分级（决定测试与上线要求）
### 3.1 风险级别
- low：文案、非核心脚本、纯展示逻辑
- mid：一般业务逻辑、非关键链路、可回滚变更
- high：订单/库存/收银/结算口径/价格更新/兼容性求解/权限安全/迁移脚本

### 3.2 规则
- risk=low：必须过“快速门禁”；可不新增测试但需说明
- risk=mid：必须新增/更新 unit 或 contract（至少其一）
- risk=high：必须：
  - unit + contract
  - 至少 1 条关键链路的集成回归（integration/E2E 任选其一，要求可复现）
  - PR 描述必须有回滚/降级方案

---

## 4. 测试策略（稳定性不牺牲速度）
### 4.1 PR 快速门禁（必须）
目标：<= 3 分钟（如超时，优先拆分）
- fmt/lint
- test-fast（unit + 关键 contract）

### 4.2 慢测试（默认合并后/夜间）
- integration / e2e / 联调
- risk=high：合并前必须跑一次慢测试（至少关键链路一条）

### 4.3 统一命令入口（为可替换性服务）
每个项目必须提供下列命令之一（实现可自由替换）：
- `make fmt` 或 `./scripts/fmt`
- `make lint` 或 `./scripts/lint`
- `make test-fast` 或 `./scripts/test_fast`
- `make test` 或 `./scripts/test_all`

---

## 5. 可替换性要求（Port/Adapter/Contract）
对以下类型依赖，默认必须“接口隔离 + 可替换实现 + 契约测试”：
- 模型/LLM 调用
- 价格/数据源抓取
- 硬件/设备
- 支付/短信/第三方 API
- 存储/缓存/队列等基础设施

最低要求：
- 定义 Port（接口）
- 每个 Adapter 在配置里可切换
- 有至少 1 组契约测试（保证替换不破行为）

---

## 6. 可观测性（快速定位问题的最低要求）
- 日志：关键路径必须包含 request_id/trace_id（或等价字段）
- 错误：区分可重试/不可重试/外部依赖错误
- 核心链路：入口、关键决策点、外部调用、落库、最终结果必须可追踪

---

## 7. 安全红线（全局）
- 禁止在代码/文档/指令/日志中出现生产密钥、token、cookie
- 外部依赖必须超时 + 限频 + 降级（特别是抓取类）
- 数据脱敏：日志不得输出敏感字段明文（按项目定义敏感字段）

---

## 8. 文档原则（省字但不丢信息）
- RFC：锁“边界/口径/验收”
- ADR：锁“为什么这么做”（避免返工时失忆）
- INS：只锁“本次改动范围/计划/验证”
- 任何“全局规则”只写 Playbook（避免 INS 重复消耗 Token）
